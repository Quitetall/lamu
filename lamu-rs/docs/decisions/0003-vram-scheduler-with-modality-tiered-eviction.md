# ADR 0003: Single-GPU NVML scheduler with modality-tiered eviction

## Status
Accepted 2026-05-31

## Context
LAMU runs on one consumer card: an NVIDIA RTX 4090 with ~24 GB of VRAM (`nvidia-smi`: 24564 MiB total). The model registry (`config/models.yaml`) is already overcommitted against that single card — it carries a 22204 MB LLM (line 38), a 17618 MB LLM (line 59), a 16000 MB `modality: tts` model (lines 131/134, the fish-speech S2-Pro server), and a 9000 MB `modality: image` model (lines 150/153, ComfyUI). No two of the large entries fit simultaneously. Backends are out-of-process `llama-server` / ComfyUI / fish-speech children, so VRAM is not allocated inside the lamu address space and cannot be measured by tracking our own allocations alone. Two further facts constrain the design: (1) other tools on this dev box — a `lamu serve` instance, `lamu-train`, a stray orphaned `llama-server` — can hold large amounts of VRAM that lamu never spawned; (2) the workload mixes modalities, where a text-to-speech or image render is a short burst against a card that is otherwise dedicated to an interactive chat LLM. The naive failure mode is letting CUDA OOM the spawn after the process is already up, which wastes the load latency and can leave a half-initialized child.

## Decision
A single in-process `VramScheduler` (`lamu-core/src/scheduler.rs`) tracks loaded models in a `HashMap<String, LoadedModel>`, queries device 0 via NVML at construction (`Nvml::init`, `scheduler.rs:22-36`) for total VRAM, and computes `available_mb()` as `total - max(sum-of-registered-models, NVML-reported-used) - reserved` (`scheduler.rs:40-51`), with `VRAM_RESERVED_MB = 1500` (`scheduler.rs:12`). Taking the max of registered-vs-actual means a foreign process that lamu did not spawn still counts against the budget. Before any load, `plan_load` (`scheduler.rs:198-212`) returns early if already loaded or if `can_fit` (`scheduler.rs:161-163`), else computes the deficit and calls `plan_eviction`, which filters out pinned and non-`Loaded` models, then sorts evictable models by `(modality.is_llm(), last_used)` so non-LLM TTS/image models (`is_llm()` false, sorts first) are reclaimed before LLMs, LRU within each tier, accumulating victims until the deficit is covered or returning empty if it cannot be met (`scheduler.rs:167-195`).

## Rationale
- VRAM is held by child processes, not by lamu, so the only ground truth is the device. NVML `memory_info().used` (`scheduler.rs:72-85`) reads the actual card; `query_gpu_pids` (`scheduler.rs:88-105`) reads per-PID usage, used post-spawn to record each model's real footprint (`loader.rs:162-169`) rather than trusting the registry's declared `vram_mb`.
- `max(registered, actual_used)` is deliberately conservative. If we trusted only our own registered sum, an orphan `llama-server` holding 22 GB would look like free space and `plan_load` would hand out a model that then OOMs. If we trusted only NVML `used`, a transient dip during a child's startup could let us double-book. The max never under-counts.
- The 24 GB ceiling is what forces tiering rather than coexistence: the 22204 MB LLM plus the 16000 MB TTS model cannot both reside on the card. Some model must leave for another to load, so eviction policy is load-bearing, not an optimization.
- Modality tiers encode the actual cost asymmetry. A TTS/image model is usually a short burst; an interactive chat LLM is the steady state. Sorting `is_llm()` ascending evicts a heavy idle TTS model ahead of a recently-used LLM, so loading the ~16 GB S2-Pro reclaims other modalities and idle LLMs first and only disturbs an active LLM when it is the last option (`scheduler.rs:175-181`).
- `pinned` (set in the registry, `types.rs:272`) gives the operator an override the LRU heuristic cannot touch — a pinned `main` model is never auto-evicted (`scheduler.rs:173`).
- `set_total_mb_for_tests` (`scheduler.rs:264-267`) nulls out NVML so `available_mb` derives entirely from registered state; without it, real GPU contention on the dev box leaks into unit tests (a live `lamu serve` holding 22 GB makes every synthetic test entry read as VRAM-exhausted).

## Alternatives Considered
- **No scheduler — let CUDA OOM.** Spawn the backend and let the driver reject the allocation. Rejected: the OOM happens after `llama-server` is already forked and partway through loading weights, so we pay full load latency to discover the failure, and the partially-initialized child must be reaped. The scheduler rejects pre-spawn with `Error::VramExhausted { need_mb, have_mb }` (`loader.rs:135-140`) before any process starts.
- **Fixed model slots.** Statically partition the 24 GB into N slots. Rejected: registry entries span 600 MB to 22204 MB (`models.yaml`); any fixed slotting either wastes most of the card on small models or cannot hold the 22 GB model at all. Bin-packing against real free VRAM uses the whole card for one big model or several small ones.
- **Naive LRU without modality tiers.** Single `last_used` ordering across all models. Rejected: a recently-used interactive chat LLM would be evicted to make room for a one-shot TTS render, then immediately reloaded — the exact thrash the `(is_llm(), last_used)` two-key sort (`scheduler.rs:181`) avoids by spending the cheap-to-reload non-LLM tier first.
- **Multi-GPU sharding / tensor parallel.** Spread one model across cards. Rejected: there is one card. NVML is queried at `device_by_index(0)` only (`scheduler.rs:25,76,92`); the scheduler has no multi-device code path. Adding it would be dead weight on this hardware.

## Consequences
- The scheduler is in-process and per-binary. `lamu-cli`, `lamu-mcp`, `lamu-api`, and `lamu-train` are separate processes that each construct their own `VramScheduler`; cross-binary arbitration is a separate concern handled by the advisory lockfile in `scheduler_lock.rs`, not by this scheduler. Two binaries that both ignore the lock can still double-book the card — the lock is advisory (`scheduler_lock.rs:25-30`).
- Eviction only ever executes on the MCP `load_model` path, which consumes the `to_evict` list, unloads each victim backend through `Backend::unload` with a 30s timeout, and sleeps 3s for VRAM to settle (`handlers.rs:394-438`). The HTTP serve path deliberately refuses to auto-evict: `ensure_loaded_with` errors with "HTTP path won't auto-evict — unload them via MCP first" when `plan_load` returns a non-empty eviction list (`loader.rs:141-147`). This couples ADR 0006 to this design — the scheduler computes evictions but only one caller is allowed to act on them.
- We are committed to NVML being present. Without it, `Nvml::init` returns None, `total_mb` is 0, `gpu_available()` is false (`scheduler.rs:55-57`), and `available_mb` returns 0 — every load is rejected as exhausted. There is no nvidia-smi subprocess fallback.
- `vram_actual_mb` is measured post-spawn from `query_gpu_pids`, but falls back to the registry's declared `vram_mb` if the PID is not found in NVML's process list (`loader.rs:166-168`). A backend whose PID NVML does not attribute (or attributes as `UsedGpuMemory::Unknown`, dropped at `scheduler.rs:101`) will be budgeted at its declared estimate, which can be wrong.
- The 3s post-eviction settle (`handlers.rs:437`) is a fixed sleep, not a poll-until-free. On a slow unload it may be too short (the new spawn races a not-yet-freed allocation) and on a fast one it is wasted latency.
- The `VRAM_RESERVED_MB = 1500` headroom is duplicated: the scheduler uses its own const (`scheduler.rs:12`) and an identical one exists in `config.rs:100`. They can drift; the scheduler's value is the one that governs the budget.

## Related Decisions
ADR 0004 (managed-subprocess backends — the children whose VRAM this scheduler measures via NVML PIDs), ADR 0006 (HTTP path never auto-evicts; eviction is an MCP-only op — directly enforced by `loader.rs:141-147` vs `handlers.rs:394-438`), ADR 0009 (confined media output — the TTS/image modalities that occupy the non-LLM eviction tier), ADR 0010 (capability + modality routing — selects which entry `plan_load` is asked to fit).

## Validation
- Right if interactive LLM sessions survive intervening TTS/image renders without being evicted and reloaded — measurable as eviction-of-LLM events in logs being rare and only occurring when no non-LLM victim exists. Wrong if logs show an active LLM evicted while an idle TTS/image model is still resident.
- Right if `plan_load` rejections (`Error::VramExhausted`) precede any CUDA OOM in a child's log. A CUDA OOM that the scheduler did not predict means `available_mb` over-reported free space — investigate whether a foreign PID was attributed by NVML.
- The `max(registered, actual_used)` guard is validated by the orphan-process case: start a foreign `llama-server` holding most of the card, then attempt a load; the scheduler must reject it. If it accepts and OOMs, the foreign-process accounting is broken.
- Revisit if the hardware changes (a second GPU, or a card large enough to hold the heavy LLM + S2-Pro simultaneously): multi-GPU would require abandoning the `device_by_index(0)`-only assumption, and a larger card would make modality tiering moot for the current registry.
