# ADR 0007: Unified cloud routing — Anthropic wire format vs OpenAI-compat for everything else

## Status
Accepted 2026-05-31

## Context
LAMU's MCP layer must reach a heterogeneous set of cloud backends: Anthropic (Claude), OpenAI, DeepSeek, MiMo, and OpenRouter as an aggregator — plus the locally hosted llama.cpp/vLLM-shaped server, which also speaks an HTTP chat API. Each backend is described by one `CloudModel` row in `~/.config/lamu/cloud-models.yaml` carrying `provider`, `base_url`, `model_id`, `api_key_env`, and an optional `chat_path` (`lamu-providers/src/cloud_config.rs:28-70`). These rows are the single source of truth shared by both the `lamu-cli` TUI and the `lamu-mcp` server (`cloud_config.rs:1-5`).

At the wire level the field is split two ways, not N ways. DeepSeek, OpenAI, MiMo, OpenRouter, Together, Groq, Zhipu, Moonshot, llama.cpp, vLLM, sglang — all speak the OpenAI `/chat/completions` shape (`messages` array, `choices[0].message.content`), as enumerated in `lamu-providers/src/payload.rs:9-13`. Only Anthropic diverges: `/v1/messages`, a top-level `system` block, a `content` array of typed blocks, `x-api-key` + `anthropic-version` headers, and its own extended-thinking + cache-control conventions. The async transport for these calls lives in `lamu-mcp`; the pure payload builders and the one env-driven header helper live in `lamu-providers` (`lamu-providers/src/lib.rs:8-11`). The decision was where to draw the dispatch seam and how many client implementations to maintain.

## Decision
`handle_cloud_query` in `lamu-mcp/src/cloud.rs` is a single function with exactly one branch point: `let is_anthropic = entry.provider == "anthropic";` (`cloud.rs:275`). When true, it builds the Anthropic-shaped request — `/v1/messages` via `entry.chat_url(&base)`, system-as-array with `cache_control: {type: "ephemeral"}`, `thinking: {type:"enabled", budget_tokens}` with the forced `temperature=1`, `x-api-key` + `anthropic-version: 2023-06-01` headers, optional `anthropic-beta` — and parses the typed-block `content` array (`cloud.rs:298-404`). For every other provider it takes the OpenAI-compat else branch: a `messages` array, top-level `thinking: {type: enabled|disabled}`, `bearer_auth` for the key, and parsing of `choices[0].message.content` plus `reasoning_content` (`cloud.rs:405-471`). OpenRouter is not a third path: it routes through the OpenAI branch and only adds its two optional attribution headers — `HTTP-Referer` and `X-Title` — gated behind `if entry.provider == "openrouter"` (`cloud.rs:434-438`). Endpoint path resolution (`/v1/messages` for anthropic, `/chat/completions` otherwise, or a verbatim `chat_path` override) is centralized in `CloudModel::chat_url` (`cloud_config.rs:105-117`). No per-vendor SDK and no external gateway process are in the call path.

## Rationale
- The N-way fan-out is a fiction. Real providers cluster into two wire formats; `payload.rs:9-13` lists nine OpenAI-compat backends covered by one builder. Dispatching on a boolean `is_anthropic` matches the actual shape of the problem, not the count of vendors.
- Provider differences that survive the format split are narrow and additive: OpenRouter needs two attribution headers (`cloud.rs:434-438`), Anthropic needs cache-control markers and a forced temperature under thinking (`cloud.rs:318-345`). These fold into the existing branch as `if` guards rather than justifying a separate code path or client.
- Routing keys on the explicit `entry.provider` field, never substring-matching the base URL, so a proxy domain containing "anthropic" can't misroute a non-Anthropic provider (`cloud.rs:271-275`). Every YAML row already declares its provider.
- The local llama.cpp server is just another OpenAI-compat row: it falls through the same else branch with no special case, so local and cloud share one transport, one timeout, one error path, and one concurrency gate (`cloud.rs:282-289`).
- Endpoint quirks are isolated in `chat_url` with a `chat_path` escape hatch (`cloud_config.rs:105-117`), so an Anthropic-format proxy at `/anthropic/v1/messages` or a vendor at `/v2/chat` is config, not code (tested at `cloud_config.rs:278-297`).
- A single function means every cloud caller — standalone `cloud_query`, the reviewer pipeline, `parallel_query`, and `warmup` — shares the same per-provider concurrency semaphore and the same `error:`-prefix convention that the ensemble/critic gates depend on (`cloud.rs:54-74`, `cloud.rs:378-383`, `cloud.rs:451-455`).

## Alternatives Considered
- **Per-provider SDK clients** — one typed client per vendor (`anthropic`, `openai`, `deepseek`, …). Rejected: nine of the supported backends share one wire format (`payload.rs:9-13`), so this would add N dependency trees and N upgrade treadmills to express a two-way split. It also breaks the OpenAI-compat long tail (Together, Groq, vLLM, llama.cpp) that has no first-party SDK but works today as a `base_url` + provider row.
- **External gateway dependency (Bifrost / LiteLLM)** — run a separate proxy process that normalizes every provider to one shape; LAMU posts only OpenAI-format to it. Rejected: it adds a managed subprocess to LAMU's deploy and failure surface for a normalization that is two `json!` literals and a header guard in-process. The `CloudModel` schema keeps vestigial "gateway-routed" support — `api_key_env: None` skips the auth header (`cloud.rs:248-261`) and `chat_url` falls back to a gateway URL when `base_url` is absent (`cloud_config.rs:105-109`) — so a gateway remains opt-in per row without being a hard dependency. The genuine downside of going direct: provider-side features a gateway would centralize (uniform quota probing, key rotation, failover) are now LAMU's problem; quota is a manual YAML field today (`cloud_config.rs:10-20`).
- **OpenAI-compat only, via an Anthropic-compatible shim** — force Claude through an OpenAI-shaped proxy and delete the Anthropic branch. Rejected: it would forfeit the features that branch encodes directly against the native API — explicit `cache_control` prompt caching (Anthropic does not auto-cache by byte prefix, unlike DeepSeek; `cloud.rs:312-324`), native extended thinking with the ≥1024 budget constraint (`cloud.rs:330-352`), and the 1M-context beta header (`cloud.rs:362-364`) — and reintroduce the external proxy this design avoids.

## Consequences
- Two payload shapes are load-bearing and must track upstream by hand. An Anthropic API change touches `cloud.rs:298-404` and `payload.rs:75+`; an OpenAI-format change touches `cloud.rs:405-471` and `payload.rs:14+`. There is no SDK absorbing breaking changes for us.
- Adding a vendor that is already OpenAI-compat is zero code: a YAML row with `provider`, `base_url`, `model_id`, `api_key_env`. Adding a third genuinely new wire format means a third branch in `handle_cloud_query` and a real refactor of the current boolean dispatch into something wider — the design is deliberately cheap for two formats and not for three.
- Provider-specific behavior lives as inline `if entry.provider == "…"` guards (OpenRouter headers at `cloud.rs:434-438`, anthropic specials throughout the true branch). This is readable at two providers but will accrete; a fourth such guard is the signal to extract a provider-capability table.
- `is_anthropic` is computed once and the function never recurses, so the shared per-provider semaphore can't self-deadlock even at cap=1 (`cloud.rs:277-289`). Every cloud path inherits one concurrency limit and one error convention — but that also means a bug in `handle_cloud_query` is a bug for all cloud callers at once.
- Gateway routing is kept as dormant, untested-at-scale support (`api_key_env: None` + `base_url`-absent fallback). It is real code that nothing in the default config exercises; it will rot unless someone runs a gateway.

## Related Decisions
ADR 0001 (MCP-first orchestration — this is the cloud transport behind those MCP tools), ADR 0002 (lean Rust backend, not a framework — direct HTTP over SDKs/gateways is the same principle applied to cloud), ADR 0008 (headless multi-model council — the ensemble reviewer calls `handle_cloud_query` per model and depends on the unified path + `error:` convention), ADR 0010 (capability + modality routing — the layer that decides which model name reaches `handle_cloud_query`).

## Validation
- Right if adding a new OpenAI-compat provider stays a YAML-only change with no edit to `cloud.rs`. The moment a new provider needs a code branch, re-examine whether the two-way split still holds.
- Right if the count of inline `if entry.provider == "…"` guards stays at two (anthropic specials + openrouter headers). A third or fourth guard means the boolean dispatch has outgrown itself; extract a capability table and revisit.
- Wrong if upstream wire-format churn (Anthropic or OpenAI) starts costing recurring manual maintenance that a maintained SDK would have absorbed — that is the signal to reconsider per-provider SDKs for the churning vendor specifically.
- Wrong if cross-provider concerns a gateway centralizes (failover, key rotation, real quota accounting) become operational pain; the dormant gateway-routed support exists precisely so adopting one is a config change, not a rewrite.
