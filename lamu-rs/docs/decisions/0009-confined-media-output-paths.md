# ADR 0009: Confine media output paths to a per-modality dir

## Status
Accepted 2026-05-31

## Context
The TTS and image MCP tools (`lamu-mcp/src/tts.rs`, `lamu-mcp/src/image.rs`) produce binary artifacts (audio, PNGs) that must land on disk so the user can open them. Each tool accepts an optional `output_path` argument so the caller can name the file. Crucially, every argument to an MCP tool is supplied by the LLM, not by a trusted human: the model decides what `output_path` to send. `tts.rs:115`, `tts.rs:311`, and `image.rs:115` all read `args["output_path"].as_str()` straight off the JSON the model emitted, and the resolved path is then handed to `std::fs::write` (`tts.rs:149`, `tts.rs:372`, `image.rs:239`). If that path were honored verbatim, a model — whether confused, jailbroken, or steered by injected content in the text it was asked to synthesize — could overwrite `~/.ssh/authorized_keys`, `~/.bashrc`, or any file the lamu process can write. The blast radius is exactly the file permissions of the MCP server process, which runs as the user. This is the same trust boundary that motivates binding to localhost (ADR 0005); here the input is the tool call itself.

## Decision
Both media tools route every write through `resolve_confined_output_path(subdir, prefix, ext, output_path)` in `lamu-mcp/src/media_paths.rs`. All output is pinned under `dirs::data_dir()/lamu/<subdir>` where `<subdir>` is a fixed literal at the call site (`"tts"` or `"images"`). A caller-supplied `output_path` may only be a non-empty relative name with no `..`: `media_paths.rs:37` rejects any value that `is_absolute()` or contains a `Component::ParentDir`, returning a plain error string instead of a path. A valid name is `dir.join(pb)`'d onto the confined dir (`media_paths.rs:43`); when no name is given, a nanosecond-stamped `<prefix>-<ns>.<ext>` is generated (`media_paths.rs:46-50`). The MCP caller therefore can choose a filename but never a directory — the tool exposes no arbitrary file-write primitive.

## Rationale
- Tool args are LLM-controlled. Treating `output_path` as trusted would convert two media tools into a general write-anywhere gadget driven by model output, including model output influenced by attacker-supplied text passed in for synthesis.
- The confinement is structural, not a blocklist: the path is *built* from a fixed base plus a sanitized leaf, so the default is "inside the dir" and an escape requires defeating the two explicit checks rather than evading a pattern match.
- Absolute-path and `..` rejection covers the two ways a relative-looking string escapes a join: an absolute path discards the base entirely, and `..` walks back up out of it. Blocking both at `media_paths.rs:37` leaves only forward descent into subfolders of the confined dir.
- The `subdir` is asserted to be a plain segment (`media_paths.rs:22-27`, `debug_assert!`). The per-call `output_path` guard runs *after* the base dir is built, so a poisoned `subdir` would escape before the guard ever sees it; the assert pins that invariant against future call sites in debug/test builds.
- `dirs::data_dir()` (falling back to `temp_dir`, never cwd — `media_paths.rs:28-29`) gives a stable, user-scoped location independent of where the server was launched, so artifacts are predictable in CI/containers.
- Nanosecond timestamps (`media_paths.rs:8`, `as_nanos`) avoid silent overwrite when two unnamed calls land in the same second.

## Alternatives Considered
- **Honor arbitrary caller paths.** Write to whatever `output_path` the model sends. Rejected: this is a direct LLM-driven arbitrary-file-write vulnerability — the model could target `/etc`, dotfiles, or the user's source tree, and a path injected via the text-to-synthesize would execute with the server's full user permissions. The cost of the alternative is a security hole, not friction.
- **Return raw bytes only, never touch disk.** The tool returns base64/bytes and the caller writes the file. Rejected: it pushes the write burden onto every caller and, for an MCP setup where the caller *is* the LLM/agent, just relocates the same write primitive somewhere with no confinement at all. It also bloats the tool-result payload (a multi-MB PNG inline) that the orchestrating model must carry in context.
- **Full sandbox / jail (bubblewrap, seccomp, separate uid).** Run the media tools in a filesystem namespace that physically cannot see anything but the confined dir. Rejected as overkill for this surface: the threat is one tool argument, and a path-resolution guard neutralizes it in ~15 lines with no process-launch, no namespace setup, and no per-platform divergence. The project already has `lamu agent` bubblewrap confinement for the genuinely dangerous case (filesystem-mutating sub-agents); a path check is the proportionate control for "the model picked a filename."

## Consequences
- Users cannot ask the model to "save the audio to ~/Desktop/note.mp3" — it lands in `<data_dir>/lamu/tts/note.mp3` and the tool reports the real path (`tts.rs:155`, `image.rs:245`). Moving it elsewhere is a manual step. This is a deliberate ergonomics-for-safety trade.
- `resolve_confined_output_path` is now load-bearing security code on the path of every TTS and image write; any new media modality MUST funnel through it rather than re-deriving its own path, and its tests (`media_paths.rs:59-79`) guard the invariant.
- The `subdir` `debug_assert!` is a no-op in release builds. It documents and tests the invariant but does not enforce it in production; safety there rests on `subdir` being a hardcoded literal at all three call sites, which a future contributor could violate.
- Residual gap, named honestly: the guard is lexical, not canonicalizing. It rejects `..` and absolute paths but does not `realpath` the result, so a symlink *already present inside* `<data_dir>/lamu/<subdir>` pointing outward would be followed by `std::fs::write`. Creating that symlink requires prior write access to the confined dir, so this is not reachable from the tool args alone, but it is not defended against either. If the confined dir ever becomes attacker-writable by another path, this matters.
- Forward subdirectories are permitted (`foo/bar.png` is accepted and `create_dir_all` will make `foo`). That is intended — it stays inside the confinement — but means the tool can create nested directories under the data dir.

## Related Decisions
ADR 0005 (bind 127.0.0.1 by default — the network trust boundary; this ADR is the tool-argument trust boundary), ADR 0004 (managed-subprocess backends with PDEATHSIG — both treat the MCP server's authority as something to scope down), ADR 0001 (MCP-first orchestration — establishes that tool callers are LLMs, which is the premise of this threat model).

## Validation
This decision is right as long as no media tool ever writes outside `<data_dir>/lamu/<subdir>` for any LLM-supplied `output_path`. Signals it should be revisited: (1) a report or test demonstrating a write escaping the confined dir — including via a pre-existing symlink — which would mean lexical guarding is insufficient and canonicalization (`std::fs::canonicalize` + prefix check) is needed; (2) a new media modality added without routing through `resolve_confined_output_path`, indicating the invariant is not being enforced by convention; (3) sustained user demand to write to chosen locations, which would require a separate explicitly-trusted path argument distinct from the LLM-controlled one rather than relaxing the guard. The unit tests at `media_paths.rs:59-79` are the regression tripwire for the core guard.
