# ADR 0010: Route by capability subset AND modality; never chat-route embedding or non-LLM models

## Status
Accepted 2026-05-31

## Context
The registry holds heterogeneous `ModelEntry` records: chat/code/reasoning LLMs, embedding models served via `llama-server --embedding`, and non-LLM modalities (fish-speech TTS, ComfyUI image). All share the same `ModelEntry` shape (`lamu-core/src/types.rs:249-294`) and live in one `HashMap<String, ModelEntry>` registry inside the `Router` (`router.rs:9-11`). The chat path — `/v1/chat/completions` and the MCP query handlers — calls `Router::route` with `capabilities: None`, which defaults the requirement to `[Capability::Chat]` (`router.rs:117-120`; callers at `lamu-api/src/openai_compat.rs:278`, `lamu-mcp/src/handlers.rs:116,299`). Selection works by capability *subset*: a candidate matches when `required.is_subset(&model_caps)` (`router.rs:228,242`).

Two entry classes break naive subset matching. (1) Embedding models intentionally carry only `Capability::Embedding` and no `Chat` (`types.rs:20-23`), so they correctly fail a `{Chat}` requirement — but they exist in the same registry and could be selected if capability semantics ever loosened. (2) Non-LLM modality entries (TTS/image) can legitimately have `capabilities: []` — see the TTS test fixture `capabilities: []` at `types.rs:588`. The subset test `required.is_subset(&empty_set)` is true exactly when `required` is empty, and a registry-wide `find` for the `main` entry or any future no-capability-filter call path would spuriously land on a 16 GB TTS server as if it were a chat model. The registry is a flat map with no built-in notion of "this thing cannot answer a chat request."

## Decision
`Router::route` first resolves aliases (`default`/`main`/`lamu` → the `main: true` entry, lowest-name tiebreak; `router.rs:49-64`), honors an explicit model name when given (`router.rs:67-114`), and otherwise selects by capability subset. The two capability-matching helpers, `find_loaded_matching` (`router.rs:219-231`) and `find_registry_matching` (`router.rs:233-245`), both apply a hard `entry.modality.is_llm()` filter *before* the `required.is_subset(&model_caps)` test. Only `Modality::Llm` entries are eligible for chat routing; TTS and image entries are unconditionally excluded (`types.rs:62-66`). Embedding models are excluded by capability (they lack `Chat`), and the `/v1/embeddings` endpoint reaches them through a separate, capability-direct path that does not use the chat router at all (`openai_compat.rs:187-205`). Among eligible candidates the router ranks loaded ones by `params_b` then `last_used` (`rank_loaded`, `router.rs:247-256`) and unloaded ones by capability count then VRAM (`router.rs:179`).

## Rationale
- Capability *subset* (not equality) lets a richer model satisfy a narrower request: a `{chat, code, reasoning}` model serves a plain `{chat}` ask. This is the correct generalization for LLMs and is why `is_subset` is used (`router.rs:228,242`).
- That same subset rule has a trap: the empty set is a subset of every set, so a `capabilities: []` entry matches any *empty* requirement. The `is_llm()` guard is the load-bearing fix — it removes TTS/image entries from the candidate pool before subset matching, so an empty-caps TTS server can never be chat-routed regardless of the requirement set (`router.rs:225,238`, comment at `router.rs:235-237`).
- Embedding models are kept out of chat routing by carrying only `Capability::Embedding` and no `Chat`, so they fail the default `{Chat}` subset test naturally (`types.rs:20-23`). They are an LLM-family backend (llama-server), so `is_llm()` alone would not exclude them — the capability vocabulary does that job, and the dedicated `/v1/embeddings` path selects them by explicit `Capability::Embedding` membership instead (`openai_compat.rs:188-195`).
- Alias resolution is deterministic: when more than one entry sets `main: true`, the lowest name wins (`min_by` on name, `router.rs:56-59`). This was a real bug fix (#22) — `HashMap` iteration order is randomized per process, so a plain `find` resolved `lamu`/`main`/`default` to different models across restarts (`router.rs:52-55`).
- A single `route` entry point serves both the HTTP compat shim and the MCP handlers, so the modality/capability guarantees hold uniformly across every front-end without per-caller re-implementation.

## Alternatives Considered
- **Name-only routing (no capability awareness)** — callers must always name an exact model; the router only resolves aliases and checks existence. Rejected: external harnesses (Claude Code, Codex, Cursor) call `/v1/chat/completions` with no model field or a generic alias and expect to land on a working chat model. Without capability awareness the no-model path (`router.rs:127-138`) and the `[Chat]` default (`router.rs:119`) could not exist, and an operator could not retarget `main` without reconfiguring every client.
- **Capability matching with no modality filter** — keep `required.is_subset(&model_caps)` but drop the `is_llm()` guard. Rejected on the concrete empty-set trap: a TTS or image entry with `capabilities: []` (the shipped TTS fixture, `types.rs:588`) satisfies `required.is_subset(&{})` whenever `required` is empty, so a future no-filter call would route a chat request to a fish-speech or ComfyUI server. The guard at `router.rs:225,238` is exactly what prevents this; removing it reintroduces the bug.
- **Explicit per-request backend pinning only** — every request carries `model: "<exact-name>"`, no capability or alias resolution. Rejected: it pushes the loaded/eviction/health logic onto every client and defeats the `main`-alias indirection (`types.rs:273-278`) that lets the operator swap the backing model centrally. The explicit-model branch already exists for callers who *want* pinning (`router.rs:67-114`); making it the only mode removes the orchestration value.

## Consequences
- The `is_llm()` calls at `router.rs:225` and `router.rs:238` are load-bearing, not cosmetic. Anyone touching modality semantics must preserve them or the empty-caps trap returns. They also appear in the VRAM eviction sort (`scheduler.rs:181`) and registry serialization skip (`registry.rs:399`), so `Modality` is now coupled across routing, eviction, and persistence.
- Embedding support lives in two places that must stay consistent: the `Capability::Embedding` cap (`types.rs:20-23`, set by the llama.cpp backend at `backends/llamacpp.rs:583`) and the standalone `/v1/embeddings` handler (`openai_compat.rs:187-264`). The chat router never touches embeddings, so a bug in the embeddings path will not surface through chat routing — and vice versa.
- Capability subset matching means a heavyweight `{chat, code, reasoning, long_context}` model will satisfy a bare `{chat}` request and, by the `params_b`-first ranking (`router.rs:251-254,179`), is preferred over a smaller chat-only model when both are loaded. This can route trivial chats to the largest model; it is intentional (prefer capability) but is a latency/VRAM cost the operator must manage via what they choose to keep loaded.
- The capability vocabulary is deliberately frozen (`types.rs:10` "Resist expanding without clear use case"). Adding a capability touches the `required_vec` glyph table (`router.rs:259-271`) and the equivalent tables in the MCP and API layers (`handlers.rs:327`, `openai_compat.rs:341`) — three sites, no single source of truth, so additions are mildly error-prone.
- Routing has no notion of cost or rate limits beyond VRAM fit and health; it picks the most-capable fitting model, full stop. Cross-vendor/cloud cost tradeoffs are handled elsewhere (see ADR 0007).

## Related Decisions
ADR 0001 (MCP-first orchestration; HTTP as a thin shim — both front-ends call the one `route`), ADR 0003 (VRAM scheduler with modality-tiered eviction — shares the same `is_llm()` modality signal), ADR 0007 (unified cloud routing — capability routing is the local counterpart), ADR 0009 (confined media output paths — the TTS/image modalities excluded here).

## Validation
- Right if: chat requests with no model field deterministically land on the `main` LLM across process restarts; embedding and TTS/image entries never appear as `model_used` for a `/v1/chat/completions` or MCP `query` call; `/v1/embeddings` resolves an embedding model independent of chat routing.
- Wrong / revisit if: a TTS or image model is observed answering a chat request (the `is_llm()` guard regressed, or a new non-LLM modality was added without updating the guard); an embedding model gets `Chat` added to its capabilities and starts being chat-routed; or the capability count grows enough that the three duplicated glyph/match tables (`router.rs:259`, `handlers.rs:327`, `openai_compat.rs:341`) drift out of sync. A regression test asserting an empty-caps non-LLM entry is excluded from `find_registry_matching` for an empty requirement would lock the guard in; its absence today is a gap.
